Metadata Schema

Every release folder must contain a release-metadata.json file. The packaging tool reads this file to determine how to package, detect, and deploy the application in Intune.

Field Required Description
applicationName Yes Display name of the application (e.g. "Nouryon Safety Suite")
anNumber Yes Nouryon AN tracking number (e.g. "AN-12345")
releaseVersion Yes Version string for this release (e.g. "2.1.0")
installerType Yes Type of installer: "EXE" or "MSI"
installCommand Yes Silent install command line (e.g. setup.exe /s)
uninstallCommand Yes Silent uninstall command line
detectionType Yes How Intune detects installation: "registry" or "msi"
registryHive Conditional Registry hive for detection (e.g. "HKLM"). Required when detectionType is "registry"
registryPath Conditional Full registry key path for detection. Required when detectionType is "registry"
registryArchitecture Conditional Registry architecture: "x64" or "x86". Required for registry detection
registryRuleType Conditional How to evaluate the registry value: "string", "version", or "integer"
registryValueName Conditional Registry value name to check (e.g. "DisplayVersion")
registryExpectedValue Conditional Expected value to match for detection (e.g. "2.1.0")
msiProductCode Conditional MSI product code GUID. Required when installerType is "MSI"
uatGroup Yes Azure AD group name for UAT assignment. "UAT-" prefix is added automatically. Group is created in Azure AD if it doesn't exist. (e.g. "NouryonSafetySuite" → "UAT-NouryonSafetySuite")
dependencies Yes Comma-separated list of dependencies in "AppName|Version" format, or "none"
supersedence Yes Comma-separated list of apps this supersedes in "AppName|Version" format, or "none"

Example: Registry Detection

Use this format when the installer writes a registry key that can be checked for version or existence. 

{
  "applicationName": "Nouryon Safety Suite",
  "anNumber": "AN-12345",
  "releaseVersion": "2.1.0",
  "installerType": "EXE",
  "installCommand": "NouryonSafetySuite-2.1.0-Setup.exe /s",
  "uninstallCommand": "NouryonSafetySuite-2.1.0-Setup.exe /s /uninstall",
  "detectionType": "registry",
  "registryHive": "HKLM",
  "registryPath": "SOFTWARE\\Nouryon\\SafetySuite",
  "registryArchitecture": "x64",
  "registryRuleType": "version",
  "registryValueName": "DisplayVersion",
  "registryExpectedValue": "2.1.0",
  "uatGroup": "UAT-NouryonSafetySuite",
  "dependencies": "none",
  "supersedence": "Nouryon Safety Suite|2.0.0"
}

Example: MSI Product Code Detection

Use this format for MSI installers where Intune can detect the app by its product code GUID. 

{
  "applicationName": "ChemTracker Desktop",
  "anNumber": "AN-67890",
  "releaseVersion": "3.0.1",
  "installerType": "MSI",
  "installCommand": "msiexec /i ChemTracker-3.0.1.msi /qn",
  "uninstallCommand": "msiexec /x {B1234567-ABCD-1234-EFGH-123456789012} /qn",
  "detectionType": "msi",
  "msiProductCode": "{B1234567-ABCD-1234-EFGH-123456789012}",
  "uatGroup": "UAT-ChemTracker",
  "dependencies": "Nouryon Safety Suite|2.1.0",
  "supersedence": "ChemTracker Desktop|2.9.0,ChemTracker Desktop|2.8.0"
}

Dependencies & Supersedence

Dependencies and supersedence relationships are identified by App Name + Version. The system resolves each reference by looking up previous packaging runs in the IntuneAppRefs table.

Format

Use the pipe-separated format AppName|Version for each reference. Separate multiple entries with commas. 

// Single dependency
"dependencies": "Nouryon Safety Suite|2.1.0"

// Multiple dependencies
"dependencies": "Nouryon Safety Suite|2.1.0,Lab Inventory Manager|1.5.0"

// No dependencies
"dependencies": "none"

// Single supersedence
"supersedence": "ChemTracker Desktop|2.9.0"

// Multiple supersedence entries
"supersedence": "ChemTracker Desktop|2.9.0,ChemTracker Desktop|2.8.0"

How resolution works: When Intune app creation runs, each AppName|Version reference is looked up in the IntuneAppRefs table (keyed by normalized app name + version). The target app must have been previously packaged and created in Intune for the relationship to be established.

Tool Path Configuration

The packaging service uses Microsoft's Win32 Content Prep Tool (IntuneWinAppUtil.exe) to convert application installers into the .intunewin format required by Intune.

Environment Variables

Variable Default Description
WIN32_PREP_TOOL_PATH None Full path to IntuneWinAppUtil.exe
WIN32_PREP_TOOL_TIMEOUT_SECONDS 300 Maximum time (seconds) to wait for the tool to complete

Tool Arguments

The tool is invoked with these arguments:

IntuneWinAppUtil.exe -c "{sourceFolder}" -s "{sourceFolder}" -o "{sourceFolder}/output" -q

Common Errors

WIN32_PREP_TOOL_PATH is not configured

The environment variable is missing or empty. Set WIN32_PREP_TOOL_PATH in local.settings.json (local dev) or in your Azure Functions app settings (production).

Tool timed out after 300 seconds

The packaging tool did not finish within the allowed time. This may happen with very large installers. Increase WIN32_PREP_TOOL_TIMEOUT_SECONDS or check the source folder for issues.

Tool exited with code 1

The Content Prep Tool returned a non-zero exit code. Check the run log for stdout/stderr output from the tool. Common causes: missing installer file, insufficient permissions, or corrupted source files.

Naming Conventions

The system normalizes application names for internal storage keys and file paths. Understanding this helps when troubleshooting or browsing storage directly.

Normalization Rules

  • Application names are converted to lowercase
  • Spaces are replaced with hyphens
  • Example: "Nouryon Safety Suite" → nouryon-safety-suite

Storage Paths

Resource Path Pattern
Run logs packaging-logs/{normalized-name}/{runId}.log
Artifacts artifacts/{normalized-name}/{runId}/{filename}.intunewin
Table partition key {normalized-name}
Table row key {version}-{runId}
